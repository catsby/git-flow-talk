There is order still in all this DVCS chaos

Introduction

This branching model is widely in use among teams using git
as their means of source control, and serve as a guide to
developing software in a managed fashion.

Truths

  • Git is decentralized
  • When working as a team, have a central "truth" repo
  • Everyone pushes and pulls from the origin
  • Can also pull from peers
  • Or just use branches on the origin

I'm the Map!

Friends!

Friends with Branches!

Main Branches

  • Origins and integrations, ho!

Master

  • Reflects production ready code
  • Every push to master is considered a new release
  • Respect ma'h authorita'h

I'm the Map!

Develop

  • Reflects on-going development for the next release
  • Often called the Integration Branch
  • This is where nightly builds come from
  • When Develop is stable enough for a release, we make our way down to Master

I'm the Map!

Supporting Branches

  • Features, Releases, Hotfixes, oh my!

Features

  • Branch from Develop
  • Merge into Develop (--no-ff)
  • Naming convention: feature/a-cool-feature
  • New hotness for an upcomming release

I'm the Map!

Releases

  • Branch from Develop
  • Merge into Develop and Master
  • Naming convention: release-x.y.z
  • Last minute updates before a new release

I'm the Map!

Hotfixes

  • Branch from Master
  • Merge into Develop and Master
  • Naming convention: hotfix-omfg-how-did-this-happen
  • Need to fix production, now

I'm the Map!
